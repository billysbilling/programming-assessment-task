Programming skill assessment task

Tic tac toe

Welcome to the programming skill assessment task, and congratulations on making it this far! Let's build a simple game.

Objective

We want to know your coding style and ability. To do this, we want you to make us a playable game of tic-tac-toe - you can implement it on the frontend or on the backend, whatever you feel comfortable with. Continue scrolling to see the requirements for whichever context you choose.

Rules

You can implement the project whichever way you see fit - however, it must be your own work. This means you are of course allowed to use external libraries, but the solution must come from you. You must mention every external source you have used (both libraries and any code snippets you have copied) and explain your reasoning behind it.

Delivery

Please deliver the completed project as a GitHub repository we have access to, complete with a nice README file. If you are not used to GitHub, don't worry, simply zip your project and send it to us, along with a readme.txt file.

Frontend

So you have chosen to implement tic-tac-toe as a frontend game. We would therefore like to see how advanced your HTML, CSS and JavaScript skills are.

The design must consist of a simple grid with Xs and Os. These can be rendered using whatever you want - if you want to flex your designing muscle you can use custom SVGs, if you want to show your use of libraries, use an icon library. We do not mind if it is the prettiest game on earth, but make it as nice as you know how to. If you know how to do CSS effects, use them! If you know an awesome library, be our guest.

We would prefer if this is implemented using a framework, if you are familiar with one. Our stack consists of React and emberjs, so either of these would be perfect. Vue and Angular also work, and if you are not familiar with a frontend framework, no worries - use jQuery or even vanilla JavaScript. But definitely use a framework if you are familiar with one. It's not the end of the world if you don't use one, so no worries.

Implementation

You can implement this game either as a 2-player game or as a player-vs-computer game. You should be able to choose who starts. There should be a status bar, we'll go into that later. When someone wins, the game should start again, and the person who didn't start the last time gets to start this time. You should keep count of who has won how many times.

Grid

Nothing too crazy, a grid where the squares are clickable when it's your turn, and where you can see the Xs and Os.

Message overlay

An overlay that is used to communicate when someone has won. It should be shown over the grid.

Status bar

A simple status bar showing whose turn it is (X or O) and how many times each has won. It should also have a button to start/continue with a new game/restart.

Backend

So you have chosen to implement tic-tac-toe as a backend game. We would therefore like to see how advanced your API service skills are.

The design must consist of an API with a few endpoints. You can write this in any language you want, using any framework you want (or none at all). We use TypeScript, PHP and Python, so using any of these is a plus - especially if you can do it in Laravel, express or Koa. If you are not familiar with a backend framework, no worries - just develop it your way, as long as it works. But definitely use a framework if you are familiar with one. It's not the end of the world if you don't use one, so no worries.

Implementation

You can implement this game either as a 2-player game or as a player-vs-computer game. All communication must be handled by endpoints, and all data must be returned as JSON.

GET /

Should return the game state. It should initialize a new game if no game has been started before.

Response body

Status code 200 OK on success, with body: 

    {
      "board": [
        [piece, piece, piece],
        [piece, piece, piece],
        [piece, piece, piece]
      ],
      "score": {
        "x": number,
        "o": number
      },
      "currentTurn": "x" | "o"
      "victory": piece
    }

Where piece is either "x", "o" or "" (an empty string) and finished is whether somebody won or the game is still going on.

POST /:piece

Should place the piece (either x or o) in the requested grid coordinates:

Request body 
    {
      "x": number,
      "y": number
    }

Where x and x can be 0, 1 or 2.

Response body

Status code 409 Conflict if a piece is being placed where a piece already is.

Status code 406 Not acceptable if a piece is being placed out of turn.

Status code 200 OK if the request succeeded, with the following response body: 

    {
      "board": [
        [piece, piece, piece],
        [piece, piece, piece],
        [piece, piece, piece]
      ],
      "score": {
        "x": number,
        "o": number
      },
      "currentTurn": "x" | "o"
      "victory": piece
    }

Where piece is either "x", "o" or "" (an empty string) and finished is whether somebody won or the game is still going on.

POST /restart

Should clear the board and update the score based on who won (the victory variable). If nobody had a victory in the previous game, it basically clears the board without touching the scores.

Response body

Status code 200 OK if the request succeeded, with the following response body: 

    {
      "board": [
        [piece, piece, piece],
        [piece, piece, piece],
        [piece, piece, piece]
      ],
      "score": {
        "x": number,
        "o": number
      },
      "currentTurn": "x" | "o",
      "victory": piece
    }

Where piece is either "x", "o" or "" (an empty string) and finished is whether somebody won or the game is still going on.

DELETE /

Should clear the board and scores.

Response body

Status code 200 OK if the request succeeded, with the following response body: 

    {
      "currentTurn": "x" | "o"
    }

Where currentTurn is either x or o, indicating who should start.
